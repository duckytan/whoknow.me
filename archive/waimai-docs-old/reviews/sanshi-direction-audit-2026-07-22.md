## 先定标尺：胡闹宇宙的核心是什么

从 whoknow.me 着陆页 + README + 决策日志 + 4 大痛点 + 产品哲学金句，抽象出**胡闹宇宙 DNA**：

```
胡闹宇宙 = 低门槛 × 搞笑 × 零代价 × 分享欲
            │         │        │         │
          美团皮     NPC戏精   不肝不氪   截图即段子
          0 学习     天然段子  零卡路里   朋友圈传播
```

### 4 条基因链（偏离一条就是变种）

| 基因 | 原文出处 | 判断标准 |
|:----|:---------|:---------|
| **① 低门槛** | 「美团 8 分相似」「打开像美团，不用学」| 用户 10 秒内能理解怎么玩 |
| **② 搞笑** | 「外卖解瘾器+段子机」「NPC 天然段子」| 核心产出是笑点 |
| **③ 零代价** | 「不肝不氪」「零卡路里」「同样的冲动，出口」| 没有疲劳感、没有负面情绪 |
| **④ 分享欲** | 「截图传播属性强」「朋友圈段子」| 用户想截屏发出去 |

### 胡闹是表现手法，不是目的

> **关键认知**：胡闹不是「越复杂越胡闹」，而是**用最简单的设计产出最意外的笑点**。
> 
> 比如：用户选地址「公厕」→ 老板台词变成「送到公厕？？」= **胡闹**（3 行代码）
> 而用户攒 35 个参数 → 触发概率公式 → 计算状态流 → 产出台词 = **复杂**，不一定是胡闹。

## 司一、明辨诀 — 朱熹「格物致知」

> 方向问题不问「好不好」，问「对不对」。

### 事实 1：剧情引擎方案 vs 4 大痛点

| 痛点 | 剧情引擎怎么解 | 评分 |
|:----|:-------------|:----:|
| **① 无聊** | 每次点单 NPC 反应不同 → 可刷内容 | 🟢 强关联 |
| **② 想笑** | 地址/备注/历史影响台词 → 天然段子 | 🟢 强关联 |
| **③ 没时间没钱** | 选菜→下单→看戏，不肝不氪 | 🟢 符合 |
| **④ 想减肥** | 点完笑完，食欲没了 | 🟢 符合 |

**结论**：引擎本身**方向没错**，4 条痛点都解了。

### 事实 2：方案里的东西，哪些对不上胡闹宇宙

| 方案内容 | 对 DNA 的匹配度 | 问题 |
|:--------|:--------------:|:-----|
| 地址/备注下拉框 | 🟢 100% | 用户 1 秒选完，简单直接 |
| NPC 台词因地址变化 | 🟢 100% | 第 1 次选「公厕」看到老板骂就会笑 |
| 权重随机 + 概率递增 | 🟢 90% | 用户感受不到机制，但感受得到不同结果 |
| 7 个剧情分支 | 🟡 70% | 好玩但触发率低，90% 用户走普通台词 |
| Flags 跨订单记忆 | 🟡 60% | **偏离「零代价」**——用户得记住了上单才行 |
| 条件字符串解析器 | 🟡 50% | 对开发者好，对用户看不见 |
| 成就收集系统 | 🔴 30% | **偏离「不肝不氪」**——成就是收集行为，有负担 |
| 社交动态信息流 | 🔴 20% | **偏离「零用户时做了就是浪费」**+ 需要多人 |
| 稀有度（网游抽卡） | 🟡 50% | 用户能感知，但「抽卡」感跟「零负担」有点冲突 |

## 司二、破妄诀 — 六祖慧能「本来无一物」

### PW-01 共识妄执：「用户越玩越深」

**我们信的**：用户会反复来点单，逐步触发所有分支，收集成就，看社交动态。

**真相可能**：
- 用户第一次来点 2-3 单，笑完了就走了
- 下次来可能是下周，已经忘了有过什么 flags
- 大部分人不会点 10 单以上

**和胡闹宇宙的矛盾**：我们默认用户是"深度玩家"，但胡闹宇宙的定位是**"零代价段子机"**——用完即走、不留存才是零代价。

### PW-12 边界妄执：「胡闹 = 越复杂越好笑」

**我们信的**：事件链越长、机制越精密，用户越觉得很胡闹。

**真相可能**：
- 用户选「公厕」→ 老板说一句「送到公厕？？」= **这是真的胡闹**
- 用户破产花 ¥300 → 触发 30% 概率 → 上吊→被救→结婚→婚礼外卖 C 位 = **这是故事**，不是胡闹

**和胡闹宇宙的矛盾**：
- 胡闹是**瞬间**的——1 秒理解，2 秒发笑
- 故事是**沉浸**的——需要注意力，需要记忆
- 如果你需要 3 段文字解释才能让人笑，那不是胡闹，是剧本

### PW-24 授权妄执：「这一步不会走偏」

**我们信的**：做了下拉框 + 引擎 + 台词，每一步都在正轨上。

**检查**：
- 第一步（台词注入地址变量）→ ✅ 100% 胡闹
- 第二步（下拉框 + 统计）→ ✅ 90% 胡闹
- 第三步（事件引擎 + 分支）→ 🟡 70% — 开始变重了
- 第四步（flags 跨订单）→ 🟡 50% — 用户可能需要"记住前情"
- 第五步（成就/稀有度/社交）→ 🔴 30% — 偏离"不肝不氪"

**风险**：这是**渐进式偏离**——每一步单看都没问题，合在一起就从「段子机」变成了「剧情游戏」。

### PW-07 时序妄执：「先搭好框架再填充内容」

**我们信的**：先把成就/社交/稀有度的 UI 框架搭好，内容以后慢慢加。

**和胡闹宇宙的矛盾**：
- 用户在空页面看到「暂无成就」「暂无动态」= **冷冰冰的骨架**
- 胡闹宇宙不应该是"框架型产品"——用户不应该看到"正在施工"
- 胡闹宇宙的核心是**内容先行**——先让用户笑，再做架子

## 司三、五行诀 — 老子「反者道之动」

### WX-01 太初诀：回到最根本

```
胡闹外卖最根本的一个动作：
  用户 → 选地址/备注 → 点菜 → 下单 → 看 NPC 演一出

这个动作最直接的"胡闹增量"是什么？
  用户选了「公厕」+ 备注「多放辣」
  → 老板：「送到公厕还要多放辣？是你口味重还是公厕口味重？」
  = 胡闹 ✅  （1 行台词就够，不需要事件链）

引擎做再大，用户感受的也只是这一句台词。
```

### WX-04 颠倒诀：最胡闹的版本 vs 最不胡闹的版本

**最胡闹的版本（方向正确）**：

```
打开 App → 选公厕 → 备注「表演才艺」→ 下单
→ 老板骂骂咧咧接单
→ 骑手路上说相声（备注触发）
→ 送达时附赠一张手写便利贴（地址触发）
→ 用户笑着截图发朋友圈：「这外卖有点东西」
```

**最不胡闹的版本（方向跑偏）**：

```
打开 App → 需要理解：成就/动态/订单 四个 Tab
→ 选菜 → 地址/备注下拉框（但没有明显变化）
→ 引擎计算了 35 个参数
→ 触发了某个分支（但台词跟普通订单差别不大）
→ 送完后解锁了一个成就（用户：成就干嘛的？）
→ 社交动态是空的（用户：没人玩？）
```

### WX-05 知几诀：现在最该做和最不该做

**最该做（最大胡闹增量，最小开发成本）**：

```
1. 地址变量 → NPC 台词（30 分钟）
   「公厕」→ 老板吐槽
   「ICU」→ 骑手小心翼翼
   「百慕大」→ 骑手迷路台词特供版

2. 备注变量 → NPC 台词（20 分钟）
   「多放辣」→ 老板：「辣死你我不管」
   「别骂了」→ 老板憋着骂
   「表演才艺」→ 骑手配送途中讲段子

3. 简单的概率偏移（10 分钟）
   地址=百慕大 → 迷路概率 +30%
   备注=多放辣 → bossMood -10
   → 不用建全量引擎，先把台词+概率的映射表落地
```

**最不该做（偏离胡闹宇宙）**：

```
1. 成就系统 UI（即使填了模板数据）
2. 社交动态信息流
3. 稀有度展示（至少先不做 UI 层）
4. Flags 系统（等台词层面的胡闹感够了再做）
```

## 📌 最终判断：方向偏离了吗？

**目前没有**，但第 1 次三司会审建议的「成就/社交/稀有度 UI」方向，如果执行了，**就会偏**。

### 偏航预警线

```
台词注入地址/备注变量  ←── 你在这里 ✅ 纯胡闹
       ↓
下拉框 + 10 个核心参数  ←── 安全，但不要扩到 35 个
       ↓
精简引擎 + 3 个分支     ←──⚠️ 注意：分支事件不要替代日常台词
       ↓
flags 原型 + 跨订单故事  ←──🔴 进入偏航区：必须保证「不记得前情也能笑」
       ↓
成就 UI + 社交 UI        ←──✋ 暂停线：没有 100 个真实订单不做
```

### 修正建议

1. **台词第一，引擎第二** — 先让选不同地址/备注的用户看到不同的 NPC 反应
2. **不做全量 35 维** — 10 个核心参数就够撑起引擎雏形
3. **成就/社交不占坑** — 不要说「先搭框架以后填充」，胡闹宇宙是内容驱动的
4. **稀有度不展示** — 后端可以有 rarity 字段，但前端不显示不宣传，怕用户当真去「刷稀有」
5. **胡闹的检验标准**：**「这句话截图发朋友圈，朋友会笑吗？」**
   - 能 → 胡闹 ✅
   - 不能 → 太重了，需要简化
