**方案中 35 个参数的收集，实际能做到多少？**

| 参数类别 | 总数 | 当前可收集 | 需要新增工量 |

### MB-04 追问性对抗

**朱熹问**：「这引擎非现在不可吗？如果只做 P1（地址/备注下拉框 + 统计），不做 P2-P6，用户会感受到什么不同？」

**答**：用户只看到地址和备注变成了下拉框，订单多了两个字段——剧情引擎不启动的话，NPC 行为跟现在一模一样。

**朱熹再问**：「那岂不是引擎做了，但用户感觉不到？」

**答**：对，引擎本身是后端逻辑，用户**只感受到 NPC 台词的变化**。如果台词库不变，改了下拉框用户也不觉得好玩。

**关键发现**：**用户在意的不是参数多精准，而是台词多离谱。**

### MB-06 推荐行动

基于事实，明辨诀建议：

```
🔴 高优 → 先把 NPC 台词改好（地址/备注影响台词），让用户有感知
🟡 中优 → 下拉框 + 统计（P1），这是引擎的输入层
🟢 低优 → 引擎框架（P2-P3）做简单版，不追求 35 维全量
⏸️ 冻结 → 成就/社交动态（P5）暂缓，等有真实用户再说
```

## 司二、破妄诀 — 六祖慧能「本来无一物」

> **核心提问**：「我们信了什么但其实未必成立？」

### PW-01 共识妄执：「用户会在乎故事」

**执**：用户点外卖的动力是看 NPC 故事。

**破**：
- 用户如果第一次打开，看到的是"确认下单"页面，不是"开始冒险"
- 用户的核心动机可能是：**这 app 能点餐**，不是**这 app 有剧情**
- 如果故事占比 > 功能，用户会觉得"花里胡哨"

**解**：NPC 故事是**佐料**，不是**主菜**。主菜永远是"能点餐→能看到配送→能收到货"。

### PW-12 边界妄执：「引擎可以覆盖所有场景」

**执**：7 个分支够覆盖大部分用户。

**破**：
- 实际触发条件很苛刻：破产救赎（¥≥300 + 30%）、暴食 ICU（≥3 单 + 40%）
- 一个用户一周可能只触发 2-3 次特殊事件
- 其余 90% 的订单走的是**普通 NPC 台词**
- 如果普通台词不好玩，特殊事件再精彩也没用

**解**：应该先保证**日常 NPC 对话的质量**（160 条模板 + 地址/备注变量注入），再优化特殊事件。

### PW-24 授权妄执：「我会自己完成这一切」

**执**：P1-P6 会由我一个人按计划完成。

**破**：
- 实际看代码量：新建 dramaEngine.ts（~180行）+ 条件解析器（~50行）+ 修改 5 个现有文件
- 但**成就 UI 页面 + 社交动态页面 + 底部 Tab 重构**全是新的 Vue 组件
- 这些 UI 工作跟剧情引擎是**两套技能树**（引擎是逻辑，UI 是组件）

**解**：引擎和 UI 可以分开做。先把引擎做出来跑通，UI 用最简方案（比如成就页面就放一个 `van-empty` 占位）。

### PW-33 最优妄执：「多分支 = 好玩法」

**执**：分支越多越好玩。

**破**：
- 人生重开模拟器的核心是**随机性 + 连锁反应**，不是分支数量
- 用户第一次触发「破产救赎」觉得 wow，第 3 次触发就"哦又来"
- 真正持久的玩法是**变量影响概率后的不确定性**，不是写死的剧本

**解**：应该把精力花在**概率计算公式的动态感**上（比如今天运气好 bossMood 高，但地址选了百慕大又拉回来），而不是写更多剧本。

## 司三、五行诀 — 老子「反者道之动」

> **核心提问**：「平衡在哪？我们偏了吗？」

### WX-01 太初诀：拆到最小

**方案拆解**：

```
剧情引擎 = 输入层 + 推演层 + 输出层

输入层:
  ├─ 35 参数收集 → 实际 P1 做 26 个就够
  └─ 条件字符串解析器 → 复用人生模拟器的，不用重写

推演层:
  ├─ DramaState 状态流 → 核心价值（状态随阶段流动）
  ├─ 权重随机选分支 → 需要
  ├─ 概率递增公式 → 你提的这个最有意思
  └─ 事件链 (next) → 需要

输出层:
  ├─ NPC 台词匹配 → 目前已有 160 条，加上地址/备注变量就够了
  ├─ 事件效果 (stateModifiers) → 需要
  ├─ 成就系统 → 现在做太早
  └─ 社交动态 → 现在做太早
```

**核心本质**：项目当前最需要的是**输入层的下单体验**（地址/备注下拉框）和**输出层的台词感知**（地址影响老板态度），**推演层可以逐步迭代**。

### WX-04 颠倒诀：怎么保证失败

**问题**：怎么让这个方案必然失败？

```
1. 一次做完 P1-P6，一个月后上线 → 用户早跑了
2. 成就/社交页面做好了，但 empty 状态放了半年 → 用户觉得半成品
3. 35 个参数全收集了，但台词还是那 160 条 → 用户感觉不到变化
4. 分支事件全是 random，用户摸不到规律 → 没策略感
5. Flags 系统做了但用户根本触发不到 → 白做
```

**反向思考后的修正**：**分批上线，每批都有用户能感知到的变化。**

```
批 1（本周）: 地址/备注下拉框 + NPC 台词扩 30 条 + 用户能看出区别
批 2（下周）: 简单版引擎（基础概率 + 不完善的 2-3 个分支）
批 3（下下周）: Flags 原型 + 1 个跨订单故事
                  ↓
              用户从第一周就开始玩
```

### WX-05 知几诀：现在做什么最对

**结合整个胡闹宇宙来看**：

```
┌─────────────────────────────────────────────┐
│ whoknow.me（品牌域名）                        │
│  ├─ whoknow-waimai（外卖·MVP 阶段）           │
│  │   ├─ 🎯 现在：让 MVP 好玩起来              │
│  │   ├─ 用户能看到地址/备注影响 NPC 对话       │
│  │   └─ 用户能感觉到"每次点单不一样"           │
│  │                                            │
│  ├─ whoknow-mart（导购·未启动）                │
│  │   └─ 剧情引擎未来可复用                     │
│  │                                            │
│  └─ whoknow-brain（大脑·空壳）                  │
│      └─ 台词/分支/成就 → 纯 JSON → brain 接管  │
└─────────────────────────────────────────────┘
```

**当前阶段最正确的 3 件事**：
1. **台词先改**——让现有 NPC 对话体现地址/备注差异（成本最低、感知最强）
2. **下拉框先上**——P1 的 UI 改动能改善下单体验
3. **引擎做最小可行版**——不做全量 35 维，先跑 10 个核心参数

## 📌 最终建议

### 保留（方案中对的）

- ✅ **权重随机 + 概率递增** — 最有价值的设计
- ✅ **条件字符串解析器** — 让分支 JSON 化，胡闹大脑可接入
- ✅ **事件链（next）** — 分支解耦，复用性强
- ✅ **DramaState 流动状态** — 核心架构，必须保留
- ✅ **Flags 系统设计** — 虽然后做，但预留了叙事连续性

### 调整（需要改的）

| 原方案 | 建议改成 |
|:------|:---------|
| P1 收集 35 参数 | P1 先收 10 个核心参数（地址/备注/金额/菜数/时段/光顾次数/今日单数/均价/历史评价/运势）|
| P2-P4 全量引擎 | P2 只跑基础版引擎（3 个分支 + 权重池 + 概率递增）|
| P5 成就+社交 UI | **暂缓到有真实用户后**，用 history.pushState 占坑就行 |
| 先搭引擎再改台词 | **先改台词再搭引擎**——地址/备注直接注入现有 160 条模板 |
| 7 个分支全做 | 先做 3 个核心分支（穷鬼、暴食、破产救赎）|

### 暂缓（不做的）

| 条目 | 理由 | 什么时候做 |
|:-----|:-----|:----------|
| 成就收集系统（含 UI） | 0 真实用户时做了是浪费 | ≥ 100 订单数据后 |
| 社交动态（含 UI） | 只有 1 个人刷的动态是灾难 | ≥ 10 活跃用户后 |
| 炸弹套餐（隐藏菜品） | 需要手动创建菜品，可延后 | P4 阶段 |
| 宿世姻缘（flags 触发） | 依赖 flags 系统，且触发门槛高 | P3-P4 阶段 |

### 建议的新计划

```
现在 ~ 今天（先改台词，最快见效）:
  把地址/备注变量注入现有 160 条 NPC 模板
  → 用户选"公厕"时，老板台词自动变"送到公厕？？"

本周（P1 精简版）:
  地址/备注下拉框 + 10 个核心参数收集
  → 用户能主动选择影响 NPC 行为

下周（P2 轻量引擎）:
  基础 DramaState + 3 个分支（穷鬼/暴食/破产）
  权重池 + 概率递增
  → 用户能感受到"同样的地址，不同概率有不同结果"

下下周（P3 扩展）:
  Flags 原型 + 跨订单故事
  日常 NPC 台词持续扩充
  → 用户能感受到"上次发生的事影响了今天"
```
