# 🍜 胡闹外卖 · 产品方案

> **版本**: v3.0 · 2026-07-22  
> **核心理念**: 分层胡闹——每个用户都能笑，但笑点不一样

---

## 一、产品定位（定稿）

### 一句话

> **披着美团外衣的段子机 + 轻量叙事游戏**

### 4 大痛点（不改）

| # | 痛 | 怎么解 |
|---|----|--------|
| ① **无聊** | 刷手机没内容 | 打开像美团 → 15 家戏精 NPC 店 → 刷老板骑手翻车 |
| ② **想笑** | 缺社交段子 | 每个操作都有回应 → 天然截图素材 |
| ③ **没时间没钱** | 但有游戏需求 | 10 秒学会 · 不肝不氪 · 点几下就有反馈 |
| ④ **想减肥** | 管不住手 | 满足点外卖的瘾 → 零卡路里 → 笑着发现被骗了 |

### 分层胡闹（核心升级）

> **不是二选一，是三层**

```
用户按自己的方式使用，体验都是完整的
                            │
        浅度用户（3 分钟）   │   深度用户（反复来）
              │              │          │
          ┌───┴──────────────┴──────────┴───┐
          │  【表层】即时段子机              │
          │  地址/备注 → NPC 直接吐槽       │← 第一单就笑
          │  不需要懂任何系统               │
          ├────────────────────────────────┤
          │  【中层】概率引擎                │
          │  参数影响概率 → 每单反应不同    │← 多试几次发现不同
          │  不告诉用户机制                 │
          ├────────────────────────────────┤
          │  【深层】跨订单叙事              │
          │  Flags + 宿世姻缘/诈尸还魂     │← 老用户的隐藏彩蛋
          │  不主动展示，偶遇式触发         │
          └────────────────────────────────┘
```

### 每层回答一个用户问题

| 层 | 用户潜意识问题 | 答案 |
|:--|:--------------|:-----|
| 表层 | "这 App 有啥好玩的？" | 选公厕→老板骂你 → 哈哈哈哈 |
| 中层 | "每次都一样吗？" | 这次老板没骂，骑手迷路了 — 跟上次不一样 |
| 深层 | "怎么我觉得老板认识我？" | "又是你？上次吃进医院的？" |

## 二、用户旅程（三层用户各自怎么走）

### 浅度用户（3 分钟 · 段子机）

```
打开 App → 首页刷 15 家店
         → 点开老王烧烤
         → 选几个菜 → 选地址「公厕」
         → 备注「多放辣」
         → 下单～～～
         ── 老板：「送到公厕还要多放辣？！」
         ── 骑手一路吐槽
         ── 送达 → 彩蛋
         → 笑着截屏发朋友圈：「这外卖有点东西」
         → 关掉，下次再来
```

### 中度用户（15 分钟 · 探索者）

```
同上 → 开始注意变化
     → 这次选「ICU」，老板变温柔了
     → 下次选「公司」，骑手嫌门禁多
     → 再试备注「表演才艺」→ 骑手路上说相声
     → 「我懂了——地址和备注会影响剧情！」
     → 开始尝试不同组合
```

### 深度用户（长期 · 叙事收集者）

```
同上 → 点够了开始注意历史
     → 破产救赎触发了结婚结局
     → 下一单发现骑手是上次那个
     → 「宿世姻缘」隐藏剧情触发
     → 「卧槽这 App 记得我？！」
     → 成为自来水传播者
```

## 三、技术实现（三层对应）

### 表层：台词注入（最简单的，先做）

```typescript
// 地址 → NPC 台词变量池
const ADDRESS_QUOTES: Record<string, string[]> = {
  home:    [],
  school:  ['学生仔啊？给你多加点料'],
  company: ['打工人啊，辛苦了'],
  toilet:  ['送到公厕？？我做的菜不配去公厕'],
  bermuda: ['百慕大？骑手你自求多福吧'],
  icu:     ['ICU 的？给你做清淡点，吃完早日康复'],
}

// 备注 → NPC 台词变量池
const REMARK_QUOTES: Record<string, string[]> = {
  more_spicy:   ['辣死你我可不管', '你也太能吃辣了'],
  less_spicy:   ['吃不了辣你来我店？'],
  no_coriander: ['又是个不吃香菜的', '香菜招你惹你了'],
  no_scold:     ['你说不骂就不骂？那我偏要骂'],
  show_time:    ['点个外卖还要才艺表演？？行吧'],
}
```

**实现成本**：30 行数据 + 20 行注入逻辑 = **第一单就能笑**

### 中层：精简引擎

**去掉 35 维全量，保留 10 个核心参数**：

| # | 参数 | 影响 |
|:-:|:----|:-----|
| 1 | `address` | bossMood ±30，迷路概率偏移 |
| 2 | `remark` | bossMood ±20 |
| 3 | `orderTotal` | 穷鬼套餐(<20) vs 破产(≥300) |
| 4 | `dishCount` | 太多→老板暴躁 |
| 5 | `timeSlot` | 凌晨/深夜 bossMood -20 |
| 6 | `shopVisitCount` | ≥3 回头客 |
| 7 | `todayOrderCount` | ≥3 暴食预警 |
| 8 | `avgDishPrice` | <10 骑手拒单 |
| 9 | `reviewTendency` | 差评多→bossMood -20 |
| 10 | `luckToday` | 每日运势调味 |

**引擎逻辑**：

```typescript
// 每个阶段开始：
// 1. 扫所有分支 → 条件匹配的加入权重池
// 2. 按权重随机选一个
// 3. 计算最终概率（baseProb + scaling）
// 4. 骰子判定 → 触发/不触发
// 5. 触发 → 执行事件链 → 更新 state → 传给下一阶段
// 6. 不触发 → 走默认 NPC 台词（含表层变量）
```

**3 个核心分支**：

| 分支 | 条件 | 设计原则 |
|:----|:-----|:---------|
| **穷鬼套餐** 🗑️ | `orderTotal < 20` | 用户第一次点就（可能）触发 |
| **暴食 ICU** 🏥 | `todayOrderCount >= 3` | 需要点 3 单后才触发，循序渐进 |
| **破产救赎** 💍 | `orderTotal >= 300` | 隐藏分支，大额订单彩蛋 |

### 深层：Flags 系统（最小版）

```typescript
// 只保留 3 个核心规则：
// 1. 分支结局→设置 flag
// 2. 下次检查 flag→触发隐藏台词
// 3. 不主动告知用户

interface UserStats {
  // ...其他字段
  flags: string[]
  // 例子: ["married_r003", "food_poison_s001"]
}

// 条件语法
"flag(married_r003)"  // 有这个标记
"!flag(haunted_boss)" // 没有这个标记
```

**深层触发链路**：

```
订单 A → 破产救赎→结婚结局 → 设 flag "married_r003"
  ↓
订单 B → 又遇到 r003（25% 概率）
       → 检查 flag → 条件匹配
       → NPC 台词变为：「好久不见……上次那顿饭我一直记得」
       → 不设新 flag，不显示成就，不弹窗
       → 就是一句话，用户自己品
```

## 四、阶段边界（做什么 + 不做什么 + 以后做）

### 现在做的

| 功能 | 优先级 |
|:----|:-----:|
| ✅ 地址下拉框 + NPC 台词注入 | P0 表层 |
| ✅ 备注下拉框 + NPC 台词注入 | P0 表层 |
| ✅ 10 个核心参数收集 | P1 中层 |
| ✅ 精简引擎（权重 + 概率 + 3 分支 + 事件链）| P1 中层 |
| ✅ 台词内嵌式成就表达（配送单/彩蛋/NPC 对话）| P1 兼顾 |
| ✅ Flags 最小原型（1 个跨订单故事）| P2 深层 |

### 暂缓的（迟早要做，不是现在）

| 功能 | 什么时候做 | 方向 |
|:----|:----------|:-----|
| ⏸️ 社交动态信息流 | ≥ 10 活跃用户后 | 先做自己高光回忆录≈朋友圈口吻 |
| ⏸️ 炸弹隐藏菜品 | 表层+中层都稳了以后 | 金锄头式搞怪分支 |
| ⏸️ 全量 35 参数 | 10 个跑通了还不够再说 | 扩参数池 |
| ⏸️ 稀有度前端展示 | 用户自发讨论稀有事件时 | 不贴标，用 NPC 台词表达 |

### 不做的（反胡闹宇宙）

| 功能 | 理由 |
|:----|:------|
| ❌ 成就 UI 系统页 | 不建空架子——成就通过 NPC 对话/配送彩蛋表达 |
| ❌ 签到/打卡 | 跟「不肝不氪」矛盾 |
| ❌ 虚拟币经济 | Phase 1 不做，无限刷 |

## 六、最终的原则

### 胡闹检验标准

每一行代码、每一条台词上线前，问这 3 个问题：

```
1. 新用户第一单能感受到吗？    ← 表层
2. 能截图发朋友圈吗？          ← 传播
3. 说明白要 3 句话以上吗？      ← 超过 3 句就是太重了
```

### 如果 3 个问题ⓛ 都回答不上来 → 不做

## 七、定位宣言（拍板用）

> **胡闹外卖不是"一个做得很复杂的外卖游戏"**  
> **而是"一个披着美团皮、选不同地址老板就会骂你、让你笑着忘了饿的外卖段子机——而且如果你够有缘，还能发现隐藏剧情"**

断句：
- 披着美团皮 = 0 学习成本
- 选不同地址老板就会骂你 = 表层胡闹
- 笑着忘了饿 = 零卡路里核心价值
- 如果你够有缘 = 深层叙事不主动展示，偶遇式触发
- 还能发现隐藏剧情 = 给探索者的红包
