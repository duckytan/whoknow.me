## 1. 过度设计审计

### ✅ 尺寸正确的部分（不是过度设计）
- **§5.1 NPC 记忆引擎（P0）**：单机 0 成本、localStorage 即实现，是脊柱，必须做。
- **§5.2 多结局矩阵**、**§5.4 戏精截图**、**§5.6 客服喜剧入口**：均复用同一套「输入→反应」数据，边际成本极低，符合「最小可行复杂度」。
- **§5.3 零卡路里（去掉破防后）**：直接服务痛点，单机 0 成本。

### ⚠️ 过度设计 1（最严重）：§6「胡闹等级 XP」+ §5.5 日常签 to「断签清零（损失厌恶）」
- **问题**：在「零负担」铁律（BRAND §1.3：不肝不氪不学）下，引入**等级成长 + 每日连续签到 + 断签惩罚**，是典型 gacha 式留存压力，与 P3 直接冲突。
- **判定**：这是整份 GDD 里唯一**正面违背设计支柱**的设计。用户选的是「戏精外卖单机版」而非 brain 长线运营，更不需要付费游戏那套管留人手段。
- **建议**：
  - 删掉「胡闹等级 XP」与任何数值化成长条；
  - 保留收集向的**图鉴 + 隐藏线 + 称号**（§5.5 后半）——这是禀赋效应/收集欲，低负担、高截图欲，符合 P2/P5；
  - 「日常」降级为**无惩罚的「今日梗」一句限定吐槽**（领了有、不领无损失），不要 streak / 不要清零 / 不要损失厌恶。

### ⚠️ 过度设计 2：§5.7 Feed「玩家相关化」作为独立 surface
- **问题**：单机无真实其他玩家，Feed 本质是把「NPC 对你订单的反应」再做一屏。而订单详情（§5.2/§5.6）已经承载了 NPC 反应——**两处重复表达同一内容**，首屏容易显得空/重复。
- **判定**：可保留，但**不是 MVP 必做**，且必须和订单详情的反应做差异化（见下）。
- **建议**：MVP 先砍 Feed 独立页；把「雷速飞赞了你的订单」这类**做成订单详情底部的一条彩蛋卡**，复用多结局数据，不单独建流。等 M3 brain 接入、有真社交素材再独立成页。

### ⚠️ 过度设计 3：§6「截图分享奖励防刷：同边框 24h 限 1」
- **问题**：单机 0 成本、无货币经济，防谁刷？这是把「有经济系统的游戏」的防作弊思维错搬到了免费玩具。
- **建议**：直接删除「防刷」逻辑，分享即得边框，无限制。

### 小结
GDD-v2 的「系统交互矩阵 §8」「配置 JSON §9」「M1–M4 清单 §11」本身是合格文档，**不属过度设计**；过度设计集中在**留存/经济层的游戏化包袱**，砍掉上面 3 处后，复杂度回到「单机喜剧玩具」的正确量级。

## 2. 胡闹宇宙理念合规审计

### ✅ 对齐良好的部分
- **BRAND 三铁律**：
  - 截图价值优先 → P2「截图即胜利」✅，且 §5.4 给了弹簧弹层 + 手写体 + 梗框的具体落地。
  - 用户习惯优先 → 仍 8 分美团皮、核心循环不变 ✅。
  - 零负担优先 → P3 声明 ✅（但被 §6 签 to 反噬，见 §1）。
- **品牌三义**（胡闹 + Who knows? + 家人们谁懂啊）→ §4/§5.5 文案语气到位。
- **四大痛点滤网**：无聊→想笑（戏精全覆盖）、管不住手（零卡路里 + 多结局复玩）均命中；「没钱也想消费」由「虚拟 0 成本随便点」天然满足。

### 🔴 红线 1（必须改）：§5.3 零卡「破防」结局 = 「说实话您胖了」
- **违反**：
  - BRAND §14.2 Do/Don't 明确禁止**「制造焦虑式营销」**；
  - `禁忌词清单-v1.0` 红灯含**外貌/体重相关焦虑、歧视**类红线；
  - 直接违背 P3 零负担（减肥痛点本是要被安抚，不是被戳）。
- **建议**：删除「破防」反向结局。零卡路里只保留**正向荒诞**（负卡路里「您摄入 -200 千卡，减肥界输了」），这是共鸣不是伤害。

### 🟡 红线 2（需标注）：竞品品牌名
- GDD 正文用「8 分相似美团」作**元描述**（设计文档内部用语，可接受），但**用户可见文案绝不能出现「美团」**。
- 关联提醒（来自本审计第 0 轮原型审计）：原型里 `美团支付 / 美团红包` 是用户可见风险点，GDD §11 重建要求里应补一句「所有用户可见文案禁竞品品牌名，用『某团/平台』代称」（对齐 `禁忌词清单` 黄灯）。
- 另有 `禁忌词` 点名 **v1 现有话术「我知道你住哪」= 威胁红线**，GDD 重建要求里应明确「复用 v1 引擎思路时，先过禁忌词清单，本句必改」。

## 3. 「3 秒理论」合规审计 ❌

### 工作定义（仓库现状）
- 仓库仅有 `whoknow-brain/docs/topics/前3秒爆点专题.md` 的**占位引用**（文件尚未写，标注「M0 后做」）。
- 我从可考证据反推工作定义：
  - BRAND 铁律「截图价值优先：每个高光交互必须回答用户会截图吗」；
  - `DEVELOPER.md` 节奏：「骑手抢单（1–3 秒）→ 气泡」；
  - 胡闹产品「刷到即笑、不笑就划走」的传播逻辑。
- **→ 工作定义**：用户进入任一关键屏 / 触发任一交互后，**必须在 3 秒内撞见第一个戏精爆点（截图欲触发点）**，否则跳出。这是比「30 秒」严苛 10 倍的硬约束。

### 当前 GDD 的缺口
- §7 新手引导写的是「进首页 **30 秒**内可见核心动词」——**基线松了 10 倍**，且没要求「首屏即爆点」。
- 全篇**没有任何一处把「3 秒首爆点」列为设计支柱红线或 broken 定义**。
- 仅 §4 Moment-to-Moment 隐含了「进店瞬间老板开场白」，但：① 那是 tap 之后、有加载延迟，未必 3 秒内；② 首页本身（未进店）没有保证 3 秒爆点。

### 修改建议（落地 3 秒理论）
1. **新增设计支柱 P6（或并入 P2）**：「**前 3 秒必有爆点**——任一关键屏进入 / 任一交互触发后 3 秒内，必须出现至少一个戏精爆点（persona 标签 / 老板开场白 / 骑手气泡 / 梗卡）。」
2. **§7 引导基线收紧**：「进首页 **3 秒**内看到带 persona 标签的商家（暴躁/哲学/懒）→ 首屏即爆点」；首单**进店 3 秒内**老板开场白必须弹出。
3. **§10 broken 定义增补**：「首页停留 >3 秒无任何戏精元素可见 → 否决」「进店 >3 秒无老板反应 → 否决」。
4. **原型对照**：prototype 首页已有 persona 标签商家，天然满足首屏 3 秒爆点——GDD 应点明这是「3 秒理论」的既有兑现点，固化下来。

## 4. 真实外卖痛点解决度审计 ✅

| 真实痛点 | v2 对应机制 | 解决度 | 备注 |
|---------|------------|-------|------|
| **等待焦虑**（骑手迟到、出餐慢） | §5.2 骑手翻车/迷路结局、§5.4 戏精化等待 | ✅ 好 | 把焦虑翻转成笑点，是胡闹最对的姿势 |
| **减肥管不住手**（四大痛点之一） | §5.3 零卡路里模式 | ✅ 好 | 自我欺骗式喜剧；**删破防后**纯正向 |
| **孤独 / 一个人吃** | §5.7「你的订单被评论/被赞」、§5.1 老板认得你 | ✅ 较好 | NPC 回应制造「被看见」；Feed 独立页可强化（见 §1 建议） |
| **售后无门 / 投诉石沉大海** | §5.6 客服喜剧入口（催单/投诉触发老板抢麦） | ✅ 好 | 把最糟的售后体验变成最佳喜剧位 |
| **花钱心疼 / 没钱也想吃** | 虚拟 0 成本随便点 | ✅ 隐含满足 | 但 §5.2「破产套餐 ¥300」类玩笑对部分人可能反向触发心疼，建议控制频次 |
| **选择困难 / 不知道点啥** | 多结局探索欲（换备注试结局） | 🟡 间接 | 未正面解决；可借「老板推荐 / 玄学点单」轻量补 |
| **商家难吃 / 踩雷** | 未直接对应 | ❌ 未覆盖 | 虚拟外卖无真实难吃风险，本就弱化，可不补 |

**结论**：v2 在「情绪类痛点」（焦虑、孤独、售后、减肥）上解决度明显优于 v1，且解法**高度贴合胡闹调性**——这是本方案最大的价值所在，应在 GDD 里把这张映射表显式写出来（目前隐含、未成文），作为「为什么做这些机制」的正当性锚点。

## 5. 重大缺口：与 DRAMA-ENGINE-V2 的对齐（方向级）

### 事实
- 仓库已有 `whoknow-waimai/docs/specs/DRAMA-ENGINE-V2.md`（2026-07-22，v2.1），定义了**「人生模拟器」式推演**：
  - 35 维输入（下单/时间/历史/关系/元参数）→ 4 阶段（接单→出餐→配送→送达）
  - 每阶段 `DramaState`（bossMood/riderMorale/delay）**跨阶段流动**
  - 骰子 + 概率 + `DramaBranch` 权重随机 + **flags 跨单叙事**（如「结婚→下次遇同骑手宿世姻缘」）
  - 台词库独立 JSON，按 `tags + moodRange` 匹配
- **我的 GDD-v2 §5.2「多结局矩阵」是一套更简化、静态的分支表**，未引用、未对齐该引擎，也未复用其 `DramaState` 流动 / `flags` 跨单机制。

### 风险判定（二选一，必须拍板）
- **情形 A**：DRAMA-ENGINE-V2 才是团队既定方向 → 我的 GDD-v2 是**倒退/错配**，应改为「前端用预制 JSON 跑 DRAMA-ENGINE 的 4 阶段 + flags，brain 后期接管概率与分支」。我的「记忆引擎」应直接落为 DRAMA 的 `flags`/历史参数，而非另写一套 `orderCount` 阈值。
- **情形 B**：用户明确选的「单机版」意图是**刻意比 DRAMA-ENGINE 更轻** → 那 GDD-v2 需**显式声明「本期不采用 DRAMA-ENGINE 的 35 维/骰子概率，仅取其 flags 跨单思路」**，并在 §1 写清取舍理由，避免后人以为漏做。

### 另外：DRAMA-ENGINE-V2 自身也有红线问题（顺带指出）
其示例分支含大量 `禁忌词清单` 红灯内容：**「上吊」「炸弹套餐→厨房爆炸进 ICU」「警察上门调查」「暴食进 ICU」「诈尸还魂」**——涉及自杀暗示、暴力/违法过程、真实医疗机构、死亡。无论采用情形 A 还是 B，**这些示例内容在落地前必须过禁忌词清单重写**（红灯 0.1% 漏掉率标准，人类稿应为 0）。本 GDD 若对齐它，需同步声明「示例仅为结构演示，话术以禁忌词清单为最终红线」。
