## 🔴 朱熹（合规视角）审查

### 发现 1 ⚠️ 「合规声明可见」条款过轻

**原文**：
> 合规声明可见（「本产品为虚拟娱乐，不涉及真实外卖或支付」）

**问题**：
- v16 升级了合规三件套（顶部 banner + 隐私政策页 + 用户协议页）
- 旧标准只要求"声明可见"，**不要求 banner/隐私页/协议页完整**
- 决策 #011 个人开发者合规策略已升级

**建议修订**：
- 升级为：「**顶部合规 banner 可见** + **隐私政策页可访问** + **用户协议页可访问** + **不接第三方追踪 SDK**」

### 发现 2 ⚠️ 缺 ICP / 主体约束条款

**问题**：
- 个人开发者主体（决策 #011）写明"免 ICP 备案 / 不内购 / 不充值"
- 旧标准**完全没有**针对主体的合规校验项
- 风险：如果后续接入广告 SDK / 第三方追踪，会撞合规

**建议新增**：
- 「无第三方追踪 SDK 代码」
- 「无支付相关 API 调用」
- 「页面无引导真实交易的文案」

## 🟢 六祖（用户体验视角）审查

### 发现 3 ⚠️ M1 框架已过时 · Day 1-7 命名误导

**原文**：
> 适用范围：M1「能下单」阶段

**问题**：
- 项目已走过 M1+M2+M3，**已经上线**
- Day 1-7 的命名不再有意义（项目不是按"天"迭代了，是按"版本"迭代 v11→v17）
- 用户现在打开 `whoknow.me/waimai/`，**验收维度应该是"完整闭环"而不是"按天逐项"**

**建议修订**：
- 重命名为"M1-M4 完整闭环验收"，删 Day 1-7 框架
- 或者：保留 Day 1-7 作为"开发期追溯"，**新增**"上线后完整验收"专章

### 发现 4 🆕 缺失 v13-v17 新功能的验收项

**问题**：v13-v17 加了大量功能，**63 项里没覆盖**：

| 版本 | 新功能 | 是否在 63 项里 |
|:----:|--------|:--------------:|
| **v13** | 无备注老板不再负面 | ❌ 无 |
| **v14** | Onboarding 3 屏引导 | ❌ 无 |
| **v15** | 评价提交庆祝态 + 分享按钮 + 催评徽标 + AI 引导 | ❌ 仅覆盖"提交评价" |
| **v16** | 合规 banner + 隐私页 + 协议页 | ⚠️ 仅 1 项弱要求 |
| **v17** | localStorage 数据埋点 7 事件 | ❌ 无 |

**建议新增**（10+ 项）：
- onboarding：3 屏滑动 / 跳过按钮 / 末屏「开始玩」写 localStorage
- 评价庆祝态：动画 / 分享按钮（native / clipboard 降级）/ 催评徽标
- 合规三件套：banner 展开跳链接 / 隐私页 4 段 / 协议页 5 段
- 数据埋点：7 个事件都能正确写入 chaos_metrics

### 发现 5 ⚠️ 「100 道菜数据完整」表述不准

**原文**：
> 100 道菜数据完整（`src/data/dishes.json`，`wc -l` 可验证行数）

**问题**：
- `wc -l` 只统计行数，**不验证数据完整性**（缺字段、错类型查不出）
- 100 道菜可能是历史目标，**实际菜品数可能已变**

**建议修订**：
- 改为「菜品数据完整（JSON 可解析 + 每条菜都有 name/price/category/id 字段）」

### 发现 6 ⚠️ 「连续下 5 单 NPC 话术不完全相同」要求过低

**问题**：
- 当前评价库 160 条模板 + 26 条老板回复 + 29 标签
- "5 单不完全相同"门槛太低，**实际跑 5 单重复率仍可能很高**

**建议修订**：
- 改为「连续下 10 单，老板话术 + 评价标签 + 评价正文 3 项中至少 2 项每次都不同」

### 发现 7 ⚠️ 缺「首次访问体验」验收

**问题**：
- v14 加了 onboarding 引导，但验收没明确"首次访问体验"
- 新用户看到的第一眼 = 决定 50% 留存

**建议新增**：
- 「**首次访问体验**：onboarding 弹窗 + 看完后下次不再弹 + 整个引导 < 30 秒」

## 🔵 老子（架构穿透视角）审查

### 发现 8 ⚠️ 「无报错」条款过于宽松

**原文**：
> 控制台无红色 Error（允许 Warning）

**问题**：
- "允许 Warning" 是 MVP 阶段思路，但**上线后 Warning 也得查**
- 慢查询、网络失败、定时器堆积都是潜在问题
- main CDP 报告就发现过：sleep 估算 vs 真实 25 秒 = 性能隐患

**建议修订**：
- 区分「致命 Error（必须修）」「资源 Warning（必须查）」「dev Warning（建议修）」
- 至少强制 0 个红色 Error + 资源 Warning ≤ 3 个

### 发现 9 ⚠️ 「性能」条款过于宽松

**原文**：
> 首页首屏加载 < 5 秒（手机 4G 网络模拟）

**问题**：
- 5 秒太长（行业标准 < 3 秒）
- 4G 网络模拟不严格，**实际国内 3G/弱网用户更多**

**建议修订**：
- 首屏 < 3 秒（4G 模拟）
- 路由切换 < 1 秒（**这是真实用户痛点**——main 报告的 25 秒是误诊，但路由切换确实应该 < 1 秒）

### 发现 10 ⚠️ 缺「数据持久化」专项验收

**原文**（Day 3-4）：
> 刷新页面后，购物车数据保留
> 订单状态持久化

**问题**：
- 分散在多个 Day，没专项验收
- localStorage 满（5MB 上限）的情况没考虑
- v17 加了 metrics 后 localStorage 占用翻倍

**建议新增**：
- 「**localStorage 健康检查**：满 / 不可用 / 多 tab 同步都有降级」

### 发现 11 🆕 缺「可分享性」专项验收

**问题**：
- 决策 #012 拍板「传播路径 A+B+D」
- v15 加了分享按钮 + 朋友圈文案
- 但 63 项里**完全没有"可分享性"专项验收**

**建议新增**：
- 分享按钮在评价完成后可见 + 点击触发 navigator.share（或 clipboard 降级）+ 分享文本格式正确

### 发现 12 ⚠️ 「真机测试」模糊

**原文**：
> 真机测试（手机浏览器打开 Vercel 链接）：可正常操作

**问题**：
- "可正常操作" 不可验证
- iOS Safari / Android Chrome / 微信内置浏览器行为差异大
- 微信内置浏览器不支持 `navigator.share`，**会触发 clipboard 降级**

**建议修订**：
- 区分浏览器：iOS Safari / Android Chrome / 微信内置
- 明确"分享降级路径在 3 种浏览器都生效"
