## 1. 为什么推翻 v1（废弃清单 + 根因）

v1 不是「皮没仿像」，是**骨架错了**：它把胡闹做成了「美团换皮 + 一堆静态戏精文案」，玩家没有任何真正的决策点，也没有任何「明天还想回来」的脊柱。

| # | v1 的问题 | 根因 | 本版怎么修 |
|---|------|------|------|
| 1 | NPC 失忆：同店第 7 单台词与第 1 单一模一样 | 没有跨单状态 | **flags + 历史参数**（§5.1，落 DRAMA 的跨单叙事） |
| 2 | 没有游玩决策：点完单就只能等 | 备注/地址是装饰，不进系统 | 备注/地址成为 **DramaBranch 触发输入**（§5.2） |
| 3 | Feed 是唯我论 | Feed 没接玩家订单 | MVP **砍 Feed 独立页**，改为订单详情彩蛋卡（§5.7） |
| 4 | 成就纯计数 | 进度无「解锁感」 | 重做为**图鉴 + 隐藏线 + 称号**（§5.5，非计数非等级） |
| 5 | 视觉粗糙、emoji 食物穿帮 | 原型用 emoji 占位 | 重建要求：真实/生成图 + 截图边框（§12） |
| 6 | 客服页孤立 | 入口缺失 | 客服成为**喜剧高潮入口**（§5.6，催单/投诉触发） |
| 7 | 新鲜感衰减无对抗 | 无日常钩子 | 无惩罚「今日梗」+ 结局图鉴（§7），今日梗由 brain `story_assets` 供料（§9.0） |

**废弃但不丢的资产**：`npcEngine.ts` 的变量注入思路、`config.json` 的 5 老板×5 骑手框架、`reviewGenerator.ts` 模板结构、`cookingTime.ts` 节奏调速——逻辑可复用，但数据层重构为支持「DramaState 流动 + flags」且对齐 brain 契约的结构（§9）。

## 2. 设计支柱（Design Pillars）

任何后续决策都拿这 6 条量，不过就砍。

| # | 支柱 | 非妥协点 | 不合格信号 |
|---|------|---------|---------|
| P1 | **每一次点单，NPC 反应都因你而不同** | 同店第 2 单 ≠ 第 1 单；写了备注 ≠ 没写 | 玩家发现「我做什么都一样」 |
| P2 | **截图即胜利** | 每个高光交互都要能回答「用户会截图发朋友圈吗？」 | 玩完一单没有任何一帧想存 |
| P3 | **零负担** | 不肝、不氪、不学；打开即玩；**无等级、无断签惩罚、无损失厌恶** | 需要看教程才懂 / 觉得「不每天来就亏了」 |
| P4 | **戏精灵魂由「选择」触发，不是贴图** | 玩家行为（备注/催单/投诉）驱动 DramaState 与分支 | 戏精只是静态文案 |
| P5 | **单机也要有「回来理由」** | 图鉴收集 + 隐藏线 + 无惩罚「今日梗」 | 次日回访率塌陷 |
| **P6** | **前 3 秒必有爆点** | 进入任一关键屏 / 触发任一交互后 **3 秒内**必须出现至少一个戏精爆点（persona 标签 / 老板开场白 / 骑手气泡 / 梗卡） | 首页停留 >3 秒无任何戏精元素；进店 >3 秒无老板反应 |

> **P6 来源**：仓库 `whoknow-brain/docs/topics/前3秒爆点专题.md` 尚未撰写，本 GDD 按 BRAND 铁律「截图价值优先」+ `DEVELOPER.md`「骑手抢单 1–3 秒」反推工作定义，**把 3 秒首爆点列为硬约束**（详见 §8 / §11）。brain 的「首页页脚水印」不计入此爆点（见 §9.5 裁定）。

## 3. 乐趣假说（Fun Hypothesis）

> **唯一必须成立的事**：玩家点完一单、看到 NPC 因为「自己写的备注 / 点的次数 / 选的玩法」而在一单里**跌宕起伏、独一无二又想截图**地反应时，会笑，并想「下次试试别的备注 / 别的店 / 别的玩法」。
>
> DRAMA-ENGINE 的因果链（差评情绪 → 出餐慢 → 骑手急 → 翻车送达）正是这条假说成立的关键——它让「不同选择 → 不同结局」是**推出来的**，不是查表查出来的。而「今日段子今日看」（brain 每日供料）让这个因果链**永不重复到底**，对抗新鲜感衰减。

如果这条不成立，其它全白搭。所有平衡优先保这条。

## 4. 核心循环文档（Core Loop）

### Moment-to-Moment（0–3 秒）
- **Action**：进首页 → **首屏即见带 persona 标签的商家**（暴躁/哲学/懒）——满足 P6「3 秒首爆点」
- **Feedback**：点进店铺瞬间，该店老板 persona 用一句戏精开场白迎接（如哲学老板：「你来了。饺子即缘分。」），≤3 秒弹出
- **Reward**：即时「这店有性格」的认知爽点 + 截图欲

### Session Loop（5–30 分钟）
- **Goal**：点 1–3 单，每单在「备注/地址/催单/投诉」上做选择，经 4 阶段推演触发不同 NPC 反应与结局
- **Tension**：零卡路里模式的「自我欺骗」拉扯（想吃 vs 想信自己没吃）；选备注的「会不会有更好笑的回应」探索欲
- **Resolution**：订单送达 + 一个**结局彩蛋**（老板金句 / 骑手翻车 / 零卡负卡路里），可截图

### Long-Term Loop（小时–周）
- **Progression**：结局**图鉴**（集齐老板线 / 骑手线 / 隐藏线）→ 解锁称号、截图边框、隐藏剧情
- **Retention Hook**：
  - **无惩罚「今日梗」**：每日一句限定吐槽（由 brain `story_assets.today_hot_topic` 供料），**领了有、不领无损失**，无 streak、无清零（对齐 P3）
  - **结局图鉴**（收集欲 / 禀赋效应：集齐 5 老板剧情线 = 「戏精宇宙 VIP」）
  - **截图分享**（社交证明：分享得限定边框，无防刷限制）

## 5. 引擎架构（对齐 DRAMA-ENGINE-V2 + brain 契约）

> **本版核心修订**：弃用 v2.0 自起的「静态多结局矩阵」，采用 `DRAMA-ENGINE-V2.md` 的**「人生模拟器」式推演**作为引擎骨架。理由：矩阵是「条件 A → 台词 B」的查表，会回到 v1「失忆 + 无因果」的死穴；DRAMA 的「变量 → 骰子+概率 → 事件 → 状态变化 → 连锁下一阶段」才真正交付 P1「因你而不同」。
> **配置来源（v2.2 新增）**：DRAMA 引擎的「变量」由两路喂入——① **玩家本地**：地址/备注/历史 flags/luckToday/phaseOfMoon/userIdHash（前端本地即可，无需 brain）；② **brain 云端**：`meta.weather`/`meta.holiday`/`meta.hot_today`（DRAMA §5.2 时间参数与 §5.5 元参数；由 brain collector 在 M0 即提供，waimai M1 即可消费，见 §9.0）。**剧本逻辑（DramaBranch）与台词池由 brain 下发 JSON（§9），前端不再硬编码分支。**

### 5.0 推演总览

```
下单（地址/备注/菜品/时间/历史 flags + brain meta）
  → DramaState（bossMood / riderMorale / delay / tags，每阶段流动）
    → 阶段1 接单（老板反应，骰子判定）
    → 阶段2 出餐（速度/态度，受 bossMood 影响）
    → 阶段3 配送（骑手/路线事件，受前序 delay 影响）
    → 阶段4 送达（结局/彩蛋，综合状态落点）
  → 不同输入 → 每次点单都是不同"剧情人生"
```

**关键区别**：
- ❌ 旧思路：`备注=公厕 → 说公厕台词`（查表，v1 死穴）
- ✅ 新思路：`备注=公厕 → bossMood -30 → 出餐慢 → 骑手急 → 一单跌宕起伏的外卖人生`（推演）

### 5.1 机制：NPC 记忆 / 跨单叙事（P0 · 全版脊柱）

**Purpose**：让 NPC 因玩家历史行为而不同地反应，根治 v1「失忆」死穴。
**Player Fantasy**：「这老板**认得我**了。」——被记住的荒诞亲密感。
**Input**：本地 `localStorage` 按 `shopId` 记录历史（对齐 DRAMA `UserStats`）：`shopVisit`（到店次数）、`dishOrders`（菜品真爱度）、`lastRemark`（上次备注）、`flags`（跨单叙事标记，如 `married_r003` / `dark_survivor_s001`）。
**Output**：历史参数进入 DramaState 初始值 + 作为 DramaBranch 的 `condition` 触发专属剧情；同时驱动 §9.1 persona 台词池的 **tier 选择**（`first`/`regular`/`vip`）。
- `shopVisit >= 3` → 回头客分支 + `regular` 台词（「又来了？缘分这东西也太不值钱了」）
- `shopVisit >= 10` → VIP 分支 + `vip` 台词（「骨灰级。这届食客，就你最长情。」）
- `flag(dark_survivor, s001)` → 旧识分支（「上回吃完三天没来的那位？」）
**Success Condition**：同店第 5 单台词与第 1 单**可观测差异**；玩家能感知「被记住」。
**Failure State**：记忆没生效 → 回到 v1 失忆 → 新鲜感 3 单内塌陷（P1 红线）。
**Edge Cases**：
- 清除缓存/换设备 → 记忆丢失 → 降级为初见模板，提示「老板把您忘了，重新认识一下？」
- `shopVisit` 溢出（>999）→ 显示「骨灰级粉丝」封顶文案
**Tuning Levers**：回头客阈值 `[待测试 3 / 5 / 10]`；每阈值文案条数 `[建议 ≥3 条随机]`。
**Dependencies**：DramaState 初始化、DramaBranch condition、确认订单备注、订单详情、图鉴。

### 5.2 机制：DramaBranch 多结局（推演落点）

**Purpose**：把「点外卖」变成「因为我的选择，得到不同结局」的游玩决策。
**Player Fantasy**：「我写这句备注，老板炸了 / 骑手翻车了 / 我负卡路里了。」
**Input**：玩家行为向量（对齐 DRAMA 参数池，见 §9）= `{addressTag, remarkTag, orderTotal, dishCount, timeSlot, shopVisit, flags, ...}` + brain `meta`（天气/节日/热搜）。
**Output**：消费 brain 下发的 `food.branches[]`（DramaBranch 数组），按 `condition`（字符串语法：`> < >= <= = != ? ! & | ()`）触发，权重随机选分支 → 阶段事件链 → 结局 phase 事件。**前端不硬编码分支，分支来自 JSON（§9.4）。**

**MVP 内置分支（话术全部过禁忌词洗稿，详见 §9.4）**：
| 分支 | 触发 condition | 类型 |
|------|---------------|------|
| 穷鬼套餐 | `orderTotal < 20` | common |
| 便宜菜多·骑手吐槽 | `avgDishPrice < 10` | uncommon |
| 破产·兄弟/恋爱 | `orderTotal >= 300` | epic（概率 0.3） |
| 暴食·老板心疼 | `todayOrderCount >= 3` | rare（概率递增） |
| 黑暗料理 | `hasTag(dark_dish)` | legendary（必触发） |
| 宿世姻缘 | `flag(married_r003) & riderId=r003` | epic（flags 触发） |
| 旧识重逢 | `flag(dark_survivor, s001) & shopId=s001` | epic（flags 触发） |

**Success Condition**：同一家店写不同备注 → 不同结局；玩家愿意「换备注再点一次」试结局。
**Failure State**：备注写了和没写结局一样 → 决策无意义 → 退回 v1（P1 红线）。
**Edge Cases**：
- 同时触发多分支 → 按权重池随机选一个（可配置 `neverExpire` 优先，如黑暗料理）
- 备注为空 → 走 `shopVisit` / `flags` 分支而非备注分支
**Tuning Levers**：`remarkTag` 分类粒度 `[待测试 5–8 类]`；每类结局文案 `[≥3 条]`；分支 `probability` / `probabilityScaling` 全部 `[待测试]`。
**Dependencies**：NPC 记忆（§5.1）、确认订单备注、订单详情、客服、图鉴。

### 5.3 机制：零卡路里模式（删破防 · 仅正向荒诞）

**Purpose**：把「想减肥管不住手」的痛点（四大痛点之一）变成自我欺骗式喜剧。
**Player Fantasy**：「我点了窑鸡，但 App 说我摄入 -200 千卡，我信了。」
**Input**：用户在设置/购物车开启「零卡路里模式」+ 填写体重（可选）。
**Output**：购物车/订单展示「预计摄入 X kcal / 本月余额 Y kcal」，按体重胡编；送达彩蛋可能触发「负卡路里」结局。
**Success Condition**：用户截图发圈「App 说我瘦了」——荒诞共鸣（对齐 P2）。
**Failure State**：变成真卡路里计算 → 丧失喜剧、引发焦虑 → 违背 P3。
**🔴 红线已修**：v2.0 的「破防：说实话您胖了」**已删除**——它踩 `禁忌词清单` 红灯（制造焦虑式营销）且违背 P3。零卡**只保留正向荒诞**（负卡路里）。
**Tuning Levers**：余额初始值 `[待测试 320 kcal]`。
**Dependencies**：购物车、订单详情、图鉴。

### 5.4 机制：戏精动态 / 截图传播点

**Purpose**：每个高光交互必须可截图（BRAND §14.4）。
**Player Fantasy**：「这一帧我要存。」
**Input**：任何结局/彩蛋/图鉴解锁瞬间。
**Output**：戏精弹层（弹簧入场 + 手写体金句 `ZCOOL KuaiLe` + 品牌锚色高亮底），自带「梗框」，一键分享。**注意：此弹层/气泡/结局卡是截图爆点，水印不覆盖其上（§9.5 裁定）。**
**Success Condition**：每单 ≥ 1 个天然截图点；首触 5 单内至少 3 次截图欲（P2 / P6）。
**Failure State**：全程无截图动机。
**Tuning Levers**：截图边框数量 `[建议首发 3 款：绿/橙/紫]`；分享奖励 `[限定边框，无防刷]`。
**Dependencies**：所有结局输出、图鉴、BRAND 锚色令牌。

### 5.5 机制：图鉴与称号（重做 · 非计数非等级）

**Purpose**：v1 纯计数成就太弱。重做为「图鉴 + 隐藏线 + 称号」的收集玩法。
**Player Fantasy**：「我把 5 个老板的剧情线全通了，我是戏精宇宙 VIP。」
**Input**：玩家行为累积（触发的 DramaBranch / 解锁的 flags / 遇到的骑手）。
**Output**：
- **图鉴**：已解锁结局/老板线/骑手，带进度条（参考原型「进行中 6」）
- **称号**：干饭哲学家、闪电催单王、凌晨干饭人、退单一姐…（原型已有，保留）
- **隐藏线**：集齐 5 老板线 → VIP 终身勋章（P2 截图点）
**Success Condition**：玩家为了「集齐」而重复点不同店/备注（良性复玩，无等级无惩罚）。
**Failure State**：仍是 `order_count >= N` 纯计数 → 无收集欲（v1 死穴）。
**Tuning Levers**：解锁阈值 `[待测试]`；称号是否带可见特效 `[建议是]`。
**Dependencies**：DramaBranch（结局收集）、NPC 记忆（flags）、客服。

> **🔴 审计修订**：v2.0 的「胡闹等级 XP + 日常签 to 断签清零（损失厌恶）」**已删除**——它违背 P3 零负担，是 gacha 式留存压力，单机喜剧玩具不需要。进度只保留「图鉴收集」，回访靠收集欲 + 无惩罚「今日梗」。

### 5.6 机制：客服对话（喜剧高潮入口）

**Purpose**：v1 客服是孤立原型。本版定为**玩家主动探索发现的喜剧奖励**。
**Player Fantasy**：「我点催单，暴躁老板抢麦回我了。」
**Input**：订单详情/我的订单的「联系商家」→ 客服页；或催单/投诉触发老板戏精回复（作为 DramaBranch 的 投诉/催单 专属分支）。
**Output**：左右气泡 + persona 标签 + 暴躁老板抢麦金句（沿用原型好评案例：「退单可以！但我的鸡已经躺在油锅里等了你 15 分钟了。它也想退单吗？」——**此句为合规喜剧示例，非威胁**）。
**Success Condition**：玩家把客服对话当彩蛋分享。
**Failure State**：客服只是普通售后流程 → 浪费最佳喜剧位。
**Tuning Levers**：触发词表 `[少送/错送/退单/催单…]` → 映射老板反应。
**Dependencies**：DramaBranch（投诉/催单分支）、订单详情。

### 5.7 机制：Feed（MVP 砍独立页 · 改为订单详情彩蛋卡）

**Purpose**：v1 Feed 只有陌生人假评价（唯我论）。本版让玩家看到**与自己订单相关**的戏精内容。
**Player Fantasy**：「雷速飞赞了我的订单：下次让我试味。」
**Input**：玩家历史订单 + flags。
**Output（MVP）**：**不做独立 Feed 页**，改为**订单详情底部一条彩蛋卡**——「骑手赞了你的订单 / 老板回复你」，复用多结局数据，不单独建流（避免与订单反应重复表达，审计 §1 过度设计 2）。
**Output（M3+）**：brain 接入、有真社交素材后，再独立成 Feed 页（展示其他玩家的稀有结局/成就）。
**Success Condition（MVP）**：玩家在订单详情感到「这个世界在回应我」。
**Failure State**：独立 Feed 与订单反应重复 → 首屏显空（已通过砍页规避）。
**Dependencies**：NPC 记忆、DramaBranch、订单详情。

## 9. 配置 JSON Schema（对齐 brain api-spec v2.1 + DRAMA-ENGINE-V2 · 禁忌词洗稿版）

> **红线**：所有 `text` 话术无论预制还是 brain 生成，**必须通过 `禁忌词清单-v1.0` 审核**（红灯 0 容忍；黄灯品牌名用「某团/平台」代称）。原 `DRAMA-ENGINE-V2.md` 示例中的「上吊 / 炸弹 / 警察上门 / ICU / 诈尸」等红灯内容**一律不得落地**——该文档自身示例需在落地前由码农虾重写。

> waimai 启动时 `GET /api/v1/food/config`，按以下 4 级降级，且**永远显示水印**（锡哥 Q23：绝不静默失败，让玩家知道 AI 在工作）。

| 降级级别 | 触发 | UI 水印文案（页脚） | waimai 行为 |
|---------|------|----------|------------|

## 11. 平衡与「坏长什么样」

**上线前必须定义 broken（先于 playtest）**：
1. **🔴 禁忌词红线漏出**：任何用户可见话术出现红灯词（自杀/暴力/违法/真实医疗机构/死亡/威胁/竞品名）→ **最高优先级否决**，立即改。
2. **记忆失效**：同店第 5 单台词与第 1 单无差异 → 立刻修（P1 红线）。
3. **决策无意义**：写备注 vs 不写备注，结局相同 → 退回 v1，否决。
4. **无截图欲**：首触 5 单内截图点 < 3 → 戏精密度不够（P2 / P6）。
5. **🔴 3 秒未爆点**：首页停留 >3 秒无任何戏精元素 / 进店 >3 秒无老板反应 → 否决（P6）。
6. **回访塌陷**：次日回访率 `[待测试基线，建议 <30% 即不合格]` → 图鉴钩子弱。
7. **焦虑反噬**：零卡路里被当真计算 → 违背 P3。
8. **🔴 配置契约断裂**：brain 下发的 JSON 缺 `branches` 或字段 shape 与本 §9 不符 → waimai 解析失败 → 触发 L3 降级（§9.5），并告警码农虾对齐（§14 D1）。

**Playtest 优先序**：先测**手感/笑率**（早期），再调平衡数值。

## 14. 对齐事项（已闭环 ✅ · 2026-07-24 直改 brain api-spec v2.2）

> 以下三条是 **v2.2 拉取 brain 最新方案后发现的契约级分歧**。锡哥授权玩法师**直接改 `whoknow-brain/docs/api-spec.md` 至 v2.2** 完成闭环（D1 补 `food.branches` / D2 改 M0 目标版本 / D3 水印裁定），无需码农虾重新设计。

### D1 · 配置契约缺 `branches` 字段（最致命）✅ 已解决
- **事实**（原）：brain api-spec v2.1 的 JSON Schema 只有 `food.boss` / `food.rider`（平铺台词），**无 `branches`**。但 DRAMA 引擎核心是 DramaBranch 条件链（§5.2），无 `branches` brain 就喂不动 DRAMA。
- **已解决**：玩法师 2026-07-24 直改 `whoknow-brain/docs/api-spec.md` 至 **v2.2**，在 §JSON Schema 的 `food` 下新增 `branches[]`（形状以本 GDD §9.4 为准）。码农虾 `generator/prompt.ts` 拼 food 模板时须输出此字段，M0 可先 1–2 条示例分支。
- **MVP 兼容**：`branches` 为空时 waimai v2 用内置 seed，brain 上线后无缝替换。

### D2 · 「waimai v18 上线」标号 vs 推翻重建 v2 ✅ 已解决
- **事实**（原）：brain api-spec 表内写「waimai（胡闹外卖）| v18 上线 | ✅ M0 接入」，暗示 brain 视旧 waimai 为已上线产品；但本 GDD 是**推翻重建 v2**（你已拍板放弃旧版）。
- **已解决**：玩法师直改 brain §服务的矩阵，M0 目标改标为「**v2 重建中（DRAMA 引擎 + brain 契约）✅ M0 接入**」。裁定：brain 的 M0 服务目标 = **新 waimai v2**（同一 §9 契约、新引擎）；旧 v1 代码（含 `/api/chaos-brain`）废弃，不迁移。

### D3 · 水印 vs 截图伪装 张力 ✅ 已解决
- **事实**（原）：brain P0-3 要求「永远显示水印（🧠 今日 AI 更新）」；waimai P2 要「8 分美团皮截图传播」。
